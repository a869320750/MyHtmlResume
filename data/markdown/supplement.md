# DevOps职业生涯规划

## 一、职业定位与目标

目标不是“多会几个工具”，而是成为能持续提升团队交付效率与稳定性的工程效能负责人。

未来 12 个月定位：

- 0-3 个月：从 CI 使用者升级为质量门禁建设者
- 3-6 个月：从流水线维护者升级为发布治理 owner
- 6-12 个月：从项目执行者升级为工程效能推动者

衡量标准：

- 交付更快：Lead Time 持续下降
- 质量更稳：Change Failure Rate 持续下降
- 过程可控：发布可追踪、可回滚、可审计

## 二、与你当前仓库对应的主战场

按现有项目分工推进，避免空谈：

- uavcli_ird：CI/CD 与发布治理主战场
- simulateflightcontrolserver：后端测试与质量门禁试点
- SIP_Guardian：桌面端打包、冒烟测试与发布规范样板

原则：先做可复制的最小闭环，再跨项目推广。

## 三、后续主要要做的事（按优先级）

### P0：先把质量门禁立起来（优先级最高）

要做什么：

- 统一三类检查：lint、test、coverage
- 设置 MR 合并门禁：pipeline 失败不可合并
- 规范失败反馈：报告可查看、问题可定位

交付物：

- 各项目 CI 配置中的 quality stage
- 测试与覆盖率报告产物
- MR 模板中的质量检查项

完成标志：

- 新 MR 在主分支全量执行质量检查
- 团队形成“先过门禁再合并”的共识与习惯

### P1：把自动化测试补到可用水平

要做什么：

- simulateflightcontrolserver：先补核心接口与 MQTT 链路单测
- 再补最小集成测试：接口 + 数据库 + 消息流
- SIP_Guardian：补启动与关键流程冒烟测试

交付物：

- UT/IT 测试目录与基础用例
- 夜间定时任务（nightly）
- 测试失败通知与处理流程

完成标志：

- MR 自动触发 UT
- nightly 自动触发 IT
- 关键功能回归可由脚本自动完成

### P2：把发布流程从“人工操作”变成“系统流程”

要做什么：

- 统一版本规则、发布说明模板、制品命名规范
- 发布产物写入元信息（提交号、构建时间、环境）
- 建立可执行回滚方案（脚本 + 文档）

交付物：

- 发布规范文档 v1
- 自动发版流程（含制品归档）
- 回滚演练记录

完成标志：

- 新版本发布全流程可追踪
- 回滚在 10 分钟内完成

### P3：建立工程效能指标体系

要做什么：

- 定义并持续统计 3 个核心指标：
  - Lead Time
  - Deploy Frequency
  - Change Failure Rate
- 每月复盘一次：问题、动作、效果

交付物：

- 指标看板（可先用 CSV + Markdown 报告）
- 月度工程效能报告

完成标志：

- 指标连续记录至少 4 周
- 至少 1 项指标有可量化改善

## 四、12 个月路线图（简版）

### 0-3 个月：打基础

- 建立质量门禁
- 补齐最小测试集
- 完成一条标准化发布链路

目标结果：从“能跑流水线”变成“能管住质量”。

### 3-6 个月：做复制

- 把试点方案复制到更多仓库
- 固化文档模板与最佳实践
- 降低团队对个人经验的依赖

目标结果：从“个人能力”变成“团队能力”。

### 6-12 个月：做治理

- 以指标驱动改进优先级
- 建立跨项目协同节奏（研发、测试、发布）
- 输出对内培训和对外可展示材料

目标结果：从“执行者”变成“工程效能 owner”。

## 五、学习路线（只学当前阶段真正需要的）

优先级建议：

1. 测试体系与质量门禁
2. Docker 与环境一致性
3. Kubernetes 部署核心概念（只学与 CI/CD 落地相关部分）

不建议：当前阶段分散精力去堆叠大量新概念。

## 六、简历与面试怎么说

简历句式：动作 + 改变 + 结果

- 主导搭建多项目 CI/CD 与质量门禁体系，覆盖构建、测试、发布全流程。
- 建立可回滚发布机制与版本追踪规范，实现发布过程可追踪、可审计、可恢复。
- 通过指标化治理持续优化交付效率，推动团队从人工交付转向系统化交付。

面试结构：背景 -> 问题 -> 行动 -> 结果

- 背景：多项目并行，交付依赖人工
- 问题：质量门禁不足、发布可追踪性弱
- 行动：分阶段推进门禁、测试、发布治理与指标体系
- 结果：交付更快、质量更稳、发布更可控

## 七、你现在立刻可以开始的三件事

1. 本周在一个主仓库落地 MR 质量门禁（lint + test + coverage）。
2. 两周内完成一个关键链路的 UT/IT 最小闭环。
3. 本月产出第一版工程效能月报（哪怕先手工统计）。

这三件事做完，你的职业叙事就会从“做过 DevOps”升级为“用 DevOps 解决过团队效率问题”。
