# ⭐ C++ 新特性常见八股文面试题

适用范围：C++11 / C++14 / C++17 / C++20 高频面试题。  
目标：不是讲全，而是优先覆盖面试最常问、最容易被追问的点。

---

# ⭐ 一、类型系统与泛型

## 1. C++ 中 any 和模板的区别是什么

### 名词定义

- 模板：编译期泛型，类型在编译期确定。
- std::any：运行期类型擦除容器，可以在同一个变量里存放不同类型的值。

### 实际答案

核心区别是：模板解决的是编译期多态，std::any 解决的是运行期“装任意类型对象”。

1. 类型确定时机不同
   - 模板：编译期确定类型，编译器会为不同类型生成不同代码。
   - any：运行期才知道内部到底装的是什么类型。

2. 性能不同
   - 模板：通常零额外抽象成本，性能最好。
   - any：有类型擦除开销，取值时还要 any_cast，性能一般不如模板。

3. 类型安全方式不同
   - 模板：编译期检查，类型不对直接编译不过。
   - any：编译期放得进去，但取的时候如果类型不匹配会抛出 bad_any_cast。

4. 适用场景不同
   - 模板：类型在接口设计阶段就能确定，追求性能和类型安全。
   - any：接口需要高度灵活，比如配置中心、事件总线、插件系统、通用参数容器。

### 一句话回答

模板是“编译期确定类型的泛型”，any 是“运行期保存任意类型的容器”；模板更高效更安全，any 更灵活但有运行期开销。

---

## 2. any 和 void* 有什么区别

### 实际答案

1. void* 本质上只是“无类型指针”，只知道地址，不知道里面是什么。
2. any 是类型安全的类型擦除容器，内部会记录真实类型信息。
3. void* 需要手动管理生命周期和类型转换，容易出错。
4. any 可以直接存对象，生命周期由对象自身管理，更现代、更安全。

### 一句话回答

void* 是不安全的裸地址，any 是带类型信息和对象语义的运行期通用容器。

---

## 3. any 和 variant 的区别是什么

### 名词定义

- std::any：可以装任意类型。
- std::variant：只能装预先声明好的几种类型之一。

### 实际答案

1. any 的类型集合是开放的，variant 的类型集合是封闭的。
2. any 更灵活，但取值依赖 any_cast，类型错误在运行期发现。
3. variant 更安全，所有可能类型在编译期就写清楚了，访问时通常配合 visit。
4. 如果类型范围已知，优先 variant；如果类型范围完全不固定，才考虑 any。

### 一句话回答

any 适合“我不知道会来什么类型”，variant 适合“我知道只会是这几种类型之一”。

---

## 4. optional、variant、any 分别适合什么场景

### 实际答案

1. optional<T>
   - 表示“有一个 T，或者没有值”。
   - 适合替代返回值里的 null、无效值、错误哨兵值。

2. variant<A, B, C>
   - 表示“是 A/B/C 中的一种”。
   - 适合状态机、表达式树、消息体等固定枚举类型集合。

3. any
   - 表示“可能是任意类型”。
   - 适合扩展性优先、运行期插件化接口。

### 一句话回答

optional 解决“有没有值”，variant 解决“是哪一种已知类型”，any 解决“运行期可能是任意类型”。

---

## 5. auto、decltype、decltype(auto) 有什么区别

### 实际答案

1. auto
   - 用初始化表达式推导变量类型。
   - 会忽略顶层 const 和引用。

2. decltype(expr)
   - 按表达式规则精确推导类型。
   - 能保留引用和 const 属性。

3. decltype(auto)
   - 让编译器按 decltype 规则推导。
   - 常用于函数返回值，尤其是想保留引用语义时。

### 面试补充

如果函数返回的是引用，而你写成 auto，可能把引用给丢了；这种场景通常要用 decltype(auto)。

---

## 6. 什么是 if constexpr

### 名词定义

if constexpr 是 C++17 引入的编译期条件分支。

### 实际答案

1. 普通 if 是运行期分支，两边代码都要先通过编译。
2. if constexpr 是编译期分支，不满足条件的分支会直接丢弃。
3. 它特别适合模板元编程，能替代很多 enable_if 的复杂写法。

### 一句话回答

if constexpr 就是“只编译成立的那个分支”，常用于模板分发和编译期条件判断。

---

## 7. 什么是 Concepts，解决了什么问题

### 名词定义

Concepts 是 C++20 引入的模板约束机制，用来描述模板参数必须满足什么能力。

### 实际答案

1. 传统模板错误信息很长，出错点往往在实例化深处。
2. Concepts 可以直接在接口层声明要求，比如“这个类型必须可比较”“必须可迭代”。
3. 好处是模板接口更清晰、报错更友好、重载选择更准确。

### 一句话回答

Concepts 就是给模板加“类型契约”，把隐式要求变成显式约束。

# ⭐ 二、对象语义与性能

## 8. 左值、右值、右值引用分别是什么

### 实际答案

1. 左值：通常有名字、可取地址、可持续存在。
2. 右值：通常是临时对象、字面量，生命周期短。
3. 右值引用 T&&：专门绑定右值，用来实现移动语义和完美转发。

### 一句话回答

右值引用的意义主要不是“多一种引用”，而是让临时对象的资源可以被安全地搬走，而不是拷贝一份。

---

## 9. std::move 和移动语义是什么关系

### 实际答案

1. std::move 本身不移动任何东西。
2. 它只是把一个对象强制转换成右值引用。
3. 真正发生移动的是后续的移动构造、移动赋值函数。
4. 对 moved-from 对象，通常只能保证“可析构、可重新赋值”，不应依赖其原值。

### 一句话回答

std::move 只是“允许移动”的标记，真正搬资源的是移动构造或移动赋值。

---

## 10. std::forward 是做什么的

### 实际答案

1. 它用于完美转发场景。
2. 当模板参数是万能引用 T&& 时，传进来的实参可能原本是左值，也可能是右值。
3. std::forward<T>(arg) 可以保留调用方原本的值类别。

### 一句话回答

move 是无脑转右值，forward 是“按原样转发”。

---

## 11. 万能引用和右值引用有什么区别

### 实际答案

1. T&& 不一定总是右值引用。
2. 只有在模板类型推导或 auto 推导的上下文里，T&& 才可能是万能引用。
3. 万能引用能同时绑定左值和右值。
4. 普通的 Widget&& 这种明确类型，就是纯右值引用。

### 一句话回答

看有没有类型推导：有推导的 T&& 多半是万能引用，没有推导的具体类型 && 就是右值引用。

---

## 12. emplace_back 和 push_back 的区别是什么

### 实际答案

1. push_back 传进去的是一个已经构造好的对象。
2. emplace_back 是在容器尾部原地构造对象。
3. 对某些场景，emplace_back 可以减少一次临时对象构造或移动。
4. 但如果传进去本来就是一个现成对象，那 emplace_back 和 push_back 优势不一定明显。

### 一句话回答

emplace_back 更适合“直接用构造参数原地构造”，push_back 更适合“我已经有一个对象了”。

---

## 13. noexcept 有什么用

### 实际答案

1. noexcept 表示函数承诺不抛异常。
2. 它既影响接口语义，也影响性能优化。
3. 标准容器在扩容时，通常更愿意使用 noexcept 的移动构造，因为这样更安全。
4. 如果移动构造不标 noexcept，某些容器可能退回到拷贝策略。

### 一句话回答

noexcept 不只是文档说明，它会直接影响标准库是否敢放心地走移动优化路径。

---

# ⭐ 三、现代资源管理

## 14. unique_ptr、shared_ptr、weak_ptr 的区别

### 实际答案

1. unique_ptr
   - 独占所有权，同一时刻只能有一个指针拥有对象。
   - 开销最小，优先使用。

2. shared_ptr
   - 共享所有权，通过引用计数管理生命周期。
   - 适合确实需要多人共享对象的场景。

3. weak_ptr
   - 不拥有对象，只是观察 shared_ptr 管理的对象。
   - 用于打破循环引用，或做缓存观察。

### 一句话回答

默认优先 unique_ptr，确实有共享需求才用 shared_ptr，遇到环就靠 weak_ptr 打断。

---

## 15. 为什么不建议滥用 shared_ptr

### 实际答案

1. 引用计数有额外性能开销。
2. 所有权边界会变模糊，容易让对象生命周期失控。
3. 一旦互相持有，就会形成循环引用，导致内存泄漏。
4. shared_ptr 适合“共享所有权”，不适合“图方便全都用”。

### 一句话回答

shared_ptr 的问题不只是慢，而是它容易把生命周期设计搞乱。

---

## 16. make_unique 和 make_shared 为什么更推荐

### 实际答案

1. 更安全，避免手写 new。
2. 异常安全更好，减少资源泄漏风险。
3. make_shared 通常还能把控制块和对象一起分配，减少一次堆分配。
4. 代码更简洁，表达“创建并交给智能指针管理”的意图更明确。

### 一句话回答

现代 C++ 基本原则就是：少写 new，优先 make_unique 和 make_shared。

---

## 17. shared_ptr 循环引用怎么产生，怎么解决

### 实际答案

1. A 持有 B 的 shared_ptr，B 又持有 A 的 shared_ptr。
2. 两边引用计数都不为 0，对象永远析构不了。
3. 解决办法是其中一边改成 weak_ptr，不参与引用计数。

### 一句话回答

shared_ptr 循环引用的本质是“引用计数闭环”，解决办法就是引入 weak_ptr 打破拥有关系。

---

# ⭐ 四、C++17 高频新特性

## 18. string_view 是什么，有什么风险

### 名词定义

std::string_view 是对一段只读字符序列的非拥有视图。

### 实际答案

1. 它不拷贝字符串，只保存指针和长度。
2. 优点是传参开销小，适合只读字符串接口。
3. 风险是它不拥有底层数据，如果原字符串销毁了，string_view 就悬空了。
4. 所以 string_view 适合“短期只读观察”，不适合随便长期保存。

### 一句话回答

string_view 最大的价值是避免拷贝，最大的风险是悬垂引用。

---

## 19. structured binding 有什么用

### 实际答案

1. C++17 支持把 pair、tuple、结构体成员拆开绑定。
2. 代码可读性更好，尤其是遍历 map 时常用。
3. 需要注意绑定方式是值还是引用，否则可能产生不必要拷贝。

### 示例

```cpp
auto [key, value] = my_map_entry;
```

---

## 20. filesystem 解决了什么问题

### 实际答案

1. 以前处理路径、遍历目录、判断文件存在，常常依赖平台 API。
2. C++17 的 std::filesystem 提供了标准化文件系统接口。
3. 可以统一做路径拼接、目录遍历、创建目录、获取文件大小等操作。

### 一句话回答

filesystem 把“跨平台文件路径和目录操作”从平台相关代码里抽出来，变成了标准库能力。

---

## 21. fold expression 是什么

### 实际答案

1. 它是 C++17 对可变参数模板的简化写法。
2. 可以把参数包按某个运算符折叠起来。
3. 常见场景是求和、拼接、批量输出。

### 示例

```cpp
template <typename... Args>
auto sum(Args... args) {
    return (... + args);
}
```

# ⭐ 五、C++20 高频新特性

## 22. span 是什么

### 名词定义

std::span 是对一段连续内存的非拥有视图。

### 实际答案

1. 它可以统一表示数组、vector、原生指针加长度。
2. 适合作为函数参数，避免传一堆 begin/end 或指针+长度。
3. 它不拥有数据，所以同样要注意底层对象生命周期。

### 一句话回答

span 可以理解成“面向连续内存的轻量只读/可写窗口”。

---

## 23. ranges 相比传统 STL 有什么改进

### 实际答案

1. ranges 把算法和视图组合得更自然。
2. 代码更偏声明式，可以链式表达过滤、转换、截取等操作。
3. 可读性更好，也减少中间容器。
4. 面试里你可以先讲思想，不一定要展开太多底层细节。

### 一句话回答

ranges 让 STL 从“算法 + 迭代器”风格，往“可组合的数据流处理”风格走了一步。

---

## 24. jthread 和 thread 有什么区别

### 实际答案

1. std::thread 析构前必须手动 join 或 detach，否则会 terminate。
2. std::jthread 是 C++20 引入的更安全线程封装。
3. 它析构时会自动 join，并支持 stop_token 协作式停止。

### 一句话回答

jthread 可以理解成“更 RAII、更安全的 thread”。

---

## 25. 三路比较运算符 <=> 是什么

### 实际答案

1. 它叫 spaceship operator。
2. 作用是统一生成比较逻辑，减少手写 <、>、<=、>=、== 的重复代码。
3. 标准库会根据成员自动推导或合成比较结果。

### 一句话回答

<=> 的核心价值是减少样板代码，让比较语义更一致。

---

# ⭐ 六、面试官爱追问的综合题

## 26. nullptr 和 NULL 的区别

### 实际答案

1. NULL 本质上通常是 0 或 ((void*)0) 这种宏。
2. nullptr 是 C++11 引入的专用空指针字面量，类型是 std::nullptr_t。
3. nullptr 能避免函数重载歧义，更类型安全。

### 一句话回答

现代 C++ 一律优先用 nullptr，不用 NULL。

---

## 27. Lambda 捕获有哪些方式

### 实际答案

1. []：不捕获
2. [=]：值捕获
3. [&]：引用捕获
4. [this]：捕获当前对象指针
5. [x, &y]：混合捕获
6. [x = std::move(obj)]：初始化捕获，C++14 很常用

### 面试补充

值捕获更安全，引用捕获更高效，但要注意生命周期；异步场景尤其要小心悬空引用。

---

## 28. constexpr 和 const 的区别

### 实际答案

1. const 表示只读，不一定是编译期常量。
2. constexpr 强调编译期可求值。
3. constexpr 可用于常量表达式场景，比如数组大小、模板参数、switch case 等。

### 一句话回答

const 是“不能改”，constexpr 是“编译期就能算出来”。

---

## 29. 为什么现代 C++ 强调 RAII

### 实际答案

1. RAII 的核心是“资源绑定对象生命周期”。
2. 对象构造时获取资源，析构时自动释放资源。
3. 这样即使发生异常，也不会轻易泄漏资源。
4. 智能指针、lock_guard、fstream 都是 RAII 思想的体现。

### 一句话回答

RAII 的价值是把资源释放从“靠人记得写”变成“对象离开作用域自动做”。

---

## 30. 面试里如果被问“你最常用哪些现代 C++ 特性”，怎么答

### 参考答案

如果是工程实践角度，我最常用的是几类：

1. 资源管理类：unique_ptr、shared_ptr、lock_guard
2. 性能与语义类：move、forward、emplace_back、noexcept
3. 泛型与类型类：auto、decltype、if constexpr、optional、variant
4. 接口优化类：string_view、span、filesystem

如果再往新一点走，我会补充 concepts 和 jthread。  
我的理解是，现代 C++ 的核心不是“语法花哨”，而是把所有权、类型安全、性能和接口表达都做得更清晰。

# ⭐ 七、速记版总结

## 一句话速记

1. 模板是编译期泛型，any 是运行期类型擦除。
2. optional 表示有没有值，variant 表示几选一，any 表示任意类型。
3. move 只是转右值，forward 才是保留值类别。
4. 默认优先 unique_ptr，共享才用 shared_ptr，循环引用靠 weak_ptr。
5. string_view 和 span 都是不拥有数据的视图，性能高但要小心悬垂。
6. if constexpr 和 concepts 是现代模板可读性的重要升级。
7. jthread 比 thread 更安全，<=> 减少比较样板代码。

---

## 建议背诵顺序

1. any / template / variant / optional
2. move / forward / 万能引用
3. unique_ptr / shared_ptr / weak_ptr
4. string_view / span
5. constexpr / if constexpr / concepts

先把这 5 组背熟，绝大部分“现代 C++ 新特性”面试追问就够用了。
