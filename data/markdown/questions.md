# DevOps / 平台工程常见面试问题与实践回答

## 1. 你设计或参与的 CI/CD 流水线复杂点在哪里？

在我参与的项目中，CI/CD 的复杂性主要体现在 **多仓协同构建、交付编排与质量门禁体系设计**，而不仅仅是单仓库自动化执行。

例如在编译器迁移项目中，需要支持 20+ 仓库并行构建与回归验证，同时保证构建结果可追溯、失败可快速定位。
我们通过流水线拆分阶段、构建结果聚合与失败粒度压缩机制，将构建失败定位从“版本级”缩短到“提交级”，显著提升研发反馈效率。

因此 CI/CD 在团队中不仅承担自动执行角色，更是研发交付体系的重要基础设施。

---

## 2. 你通常如何设计 GitLab CI 流水线结构？

一般会按照交付路径拆分为：

* prepare（环境准备 / 依赖校验）
* build（并行编译 / 缓存加速）
* test（单元测试 / 自动化验证）
* package（制品打包 / artifact 管理）
* release（版本发布 / 推送仓库）

同时会重点关注：

* job 并行策略与 stage 编排
* artifact 生命周期与传递机制
* 构建缓存策略（减少重复编译）
* 快速失败机制（fail-fast）
* 多仓触发与 parent-child pipeline 编排

目标是构建 **可扩展、可复用、可观测的流水线体系**。

---

## 3. 如何做质量门禁？

质量门禁不仅仅是 CI 状态是否成功，而是将 **关键质量指标工程化纳入合入条件**。

例如：

* 编译成功率
* 静态检查结果
* 核心自动化测试通过率
* 构建性能指标
* 补丁场景强制回归流水线

通过这些门禁策略，可以实现：

* 风险前置
* 主干稳定性提升
* 发布质量可控

## 4. Docker 在你项目中主要用于什么？

Docker 在我的实践中主要用于 **构建环境标准化与测试环境快速拉起**。

例如：

* 固化编译器版本与工具链
* 封装依赖库与系统配置
* 通过镜像版本实现构建环境可追溯
* 快速构建一致的测试环境

在复杂嵌入式或驱动相关场景中，Docker 主要解决用户态依赖一致性问题；涉及内核或硬件差异时，则需要结合专用测试环境或环境基线管理。

---

## 5. 如何解决环境不一致问题？

通常采用多层策略：

1. 容器化构建环境（解决工具链与依赖差异）
2. 环境版本基线管理（控制 OS / 内核版本）
3. 自动化部署校验机制（启动前环境自检）
4. 制品可追溯与构建记录归档

通过这些措施，可以显著降低“环境问题导致交付失败”的概率。

---

## 6. 如何推动流程或平台在团队落地？

我的做法一般是：

* 选择试点项目快速验证收益
* 用数据证明改进价值（构建时间 / 发布失败率 / 人工成本）
* 编写使用文档与最佳实践
* 组织培训与答疑
* 建立流程指标持续跟踪

流程落地的关键不是工具，而是 **建立团队对工程化交付的共识**。

---

## 7. 构建失败如何处理？

在工程体系中，主干构建畅通是最高优先级问题之一。

通常会：

* 对失败类型进行分类统计（依赖变更 / 脚本问题 / 代码缺陷）
* 快速恢复流水线可用状态
* 建立失败告警与责任归属机制
* 推动长期稳定性优化（缓存策略 / 编排优化）

目标是降低研发阻塞时间，提高整体交付节奏。

---

## 8. 你理解的平台工程价值是什么？

平台工程的核心价值在于：

* 提升研发效率
* 降低交付不确定性
* 将经验沉淀为可复用基础设施
* 支撑多产品线规模化研发

本质上是通过工程化手段，让团队从“依赖个人能力”转向“依赖体系能力”。
