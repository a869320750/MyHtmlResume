# 问题发现与buglist
先说补丁版本交付，我的理解是这样的，华为的维护流程还是比较完善的。我们有一个buglist的平台。这个平台上面会记录所有我们认为会影响现网正在跑的业务的潜在风险bug。主要来源是：
1、一些低概率现网问题，开发灭火了之后会对这个现网问题进行回溯分析，分析的结果就会作为buglist数据留下来。
2、在研测试过程中发现的问题，在开发定位问题的时候，会被要求分析影响范围，如果是之前版本就有的bug，也会填写buglist数据存下来（这里有一个机制，如果开发不认真分析，现网出现问题了之后，发现家里测出来过，但是开发没有提buglist，开发会担责任，我觉得这个机制非常的合理）
3、开发在梳理自己的代码的时候，可能会自己意识到一些bug，这时候也可以自提单自提buglist，这一步是会受到鼓励和奖励的。（至少我的权力范围内我会给他奖励，或者在给他评绩效的时候我会提到这一点）

# 补丁范围筛选
有了buglist之后呢，我们的目标就是要把buglist都做成补丁，通过补丁版本发出去，但是每个补丁版本能承载的补丁数量是有限的，所以说我们就要对补丁做优先级的排布和筛选，对于一些影响比较大的，触发概率比较高的问题，就得尽可能的合入到最近的补丁版本里面。我首先会和我们的维护的系统工程师SE去商讨一下，我们的哪些buglist需要优先处理，哪些往后排。我们维护组内部先优先做出一个筛选建议。然后我会去找各个资源组的PL（资源主管一般被称之PL），和他讨论他们组需要交付的补丁，这其中难免会有些人不重视这件事情，或者以人力不够为理由推辞。我就会晓之以理动之以情，和他们讲解一下这个东西概率有多高，如果在外网爆了，对他们的影响是什么。让他们自己就主动想做这件事。反正我是不喜欢死命压他们，最后实在是不想做就算了。我把风险报给大领导说一下他裁决了确实没人力就算了。好在大家都比较配合，在我讲了一番道理自后都还是愿意配合的。我们会拉上各个PL参加补丁版本的例会，在会议上面我们的议题就是向版本的领导建议，我们要制作这些补丁，这些补丁的价值在哪里。然后最终会由版本的SE和TSE裁决要不要落入这个版本，进行最终锁定。



# 制定补丁计划
规划好了范围之后，我们就进入了计划制订阶段，这个通常是我来负责的，我会根据他们的情况以及补丁版本的最终发布日期给这些开发同学排交付计划（反正我排布计划都是比较保守的，尽量不让他们加班），这个计划分几个部分：
1、开发方案评审阶段，开发需要了解这个buglist的来龙去脉，当时发生了什么问题，这个问题怎么回事（因为提buglist的人和开发补丁的人基本不太会是一个人），然后根据这些内容去设计他的方案，这些方案是需要拉上我们基础设施的SE，TSE，平台的SE，TSE一起来评审的。这四位系统工程师我是非常佩服的，能从非常顶层的角度去和这些个开发去沟通。给出建议。
2、代码开发阶段，这个阶段分两部分，业务代码开发和二进制替换脚本，这个阶段基本上只有一天，因为方案本身都不会太复杂，就是几行的修改。（因为我们分布式是多进程的，所以说我们的代码基本上都是二进制替换类型的补丁，不涉及那种hotpatch，我们集中式的那种appdata就得hotpatch了，我了解了一下基本原理是通过偏移量，把二进制里面的某个函数直接替换成我们的新函数，贼麻烦。）
3、开发自验证阶段，开发完了基本上所有补丁流程都得走一圈，确保可用。这一步耗时相对比较长要保证测试结果可信。
4、测试串讲阶段，要把测试拉到会上讲一遍方案，测试同学就会以测试的角度出测试方案，然后反串讲他们的测试策略。这个地方也需要内部和平台的两位TSE来参与。
5、代码合入阶段，测试测完了没问题之后，就要走代码合入了，这一步需要这个模块最权威的DE（设计工程师），几位TSE一起参与。确保最后检视一遍代码，进一步减小代码发布出去之后的风险。
这五个步骤结束之后，一个补丁就算是做完了。


# 执行阶段

这一部分就是正常的按照计划来开展就行了，我觉得这里面有一个非常好的实践，就是统一的在线文档链接。我们搞了一个在线的word文档链接，每个补丁对应一个，所有人相关都可以直接通过链接打开，查看。这个文档的每一章都有价值，补丁背景，补丁方案，补丁流程图，补丁评审会议的纪要截图，测试串讲纪要截图，代码检视纪要截图，开发自验证的流程截图，开发给测试的建议。直接都在一个文档里面，我作为subPM的最终工作就是要审核这个报告的完整性。这里面的文档基本上都是我创建的，然后我也有修改权限。

我的大部分工作就是看这这帮开发做这个工作，然后协调各个相关的SE和TSE来参加会议，所以我每天的大部分时间都在拉会和约会。与此同时，我还会主动的帮开发减少一些工作内容，比如说上面文档里面的截图，大部分就都是我开完会，写了评审纪要之后，我直接截图填在文档里面，因为这个属于流程上面的事情，可以让开发同学投入更多的时间多思考一下测一下。
